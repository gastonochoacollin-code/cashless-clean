B) Analisis funcional (alto nivel)

Sesion y tenant: cashless.session en localStorage define operatorId, token, tenantId. Headers base en common.js: X-Operator-Id, X-Operator-Token, X-Tenant-Id, Authorization.
Festival: se guarda en cashless.festivalId y se envia como X-Festival-Id para clamp en reportes/ventas.
NFC / UID: lector publica en /api/uid y lectura consume en /api/last-uid con terminalId (aislamiento por PC).
Ventas: POS usa areas/productos y cobra con /api/charge-v2; inventarios se cruzan contra ventas reales por SaleItems.
Recargas: caja usa /api/topups (o fallback legacy) con paymentMethod; cortes y reportes en /dashboard-caja/*.
Inventarios: ya existe backend dedicado con /api/inventory/*; el estado vive en base real por tenant y se mueve por entradas, transferencias y ventas.
Reportes: /reports.html define filtros globales y /reportes-barra.html cubre la vista operativa de barra.

B.1) README rapido de lectores

Cada lector debe correr apuntando al backend y a una terminal unica. El tercer parametro es el terminalId que aisla la lectura por PC.
Formato: dotnet run -- "BASE_URL" "/api/uid" "TERMINAL_ID"
EscenarioComando
Lector barra dotnet run -- "http://IP_DEL_SERVIDOR:5001/" "/api/uid" "BARRA-02"
Lector barra 1 dotnet run -- "http://IP_DEL_SERVIDOR:5001/" "/api/uid" "BARRA-01"
Lector comida dotnet run -- "http://IP_DEL_SERVIDOR:5001/" "/api/uid" "COMIDA-01"
Regla operativa: la terminal configurada en el lector debe coincidir con la terminal guardada en la pantalla web de esa misma PC.
Ejemplo: si el lector corre con "COMIDA-01", en /pos.html o /recargas.html de esa PC debes seleccionar COMIDA-01.
Si no se manda terminalId: el backend responde terminalId_required para evitar mezclar lecturas entre equipos.
